# Документация инструмента: file_ops.py (Работа с файлами)

> **Для вызова у модели:** `read_file`, `list_files`, `edit_file`, `write_file`, `search_in_files` и связанные операции остаются **отдельными** тулами (не `compact_tools`). См. [COMPACT_TOOLS.md](../COMPACT_TOOLS.md).

Этот инструмент является основой для работы агента с файловой системой. Он позволяет читать, записывать, изменять и искать файлы.

## Для всех: Что это делает?
Представьте, что у агента есть руки, которыми он может открывать папки, читать документы, вносить в них правки карандашом или полностью переписывать страницы на новые. 
- **`read_file`**: Открыть файл и прочитать текст.
- **`list_files`**: Посмотреть список файлов в папке (как в Проводнике или Finder).
- **`search_in_files`**: Найти конкретное слово или фразу во всех файлах сразу.
- **`edit_file`**: Заменить один кусок текста на другой (как "Найти и заменить").
- **`write_file`**: Создать новый файл или полностью заменить старый новым текстом.
- **`get_file_line_count`**: Быстро узнать, сколько строк в файле.

## Для разработчиков: Разбор кода (подробно)

### Зависимости и утилиты (Строки 1–34)
- **Строки 1–4**: Импорты стандартных библиотек (`os`, `pathlib`) для работы с путями.
- **Строка 6**: Импорт `@tool` из LangChain для регистрации функций как инструментов агента.
- **Строки 8–17**: Попытка импортировать локальные утилиты `resolve_abs_path` (безопасное разрешение путей) и `save_version` (автоматическое сохранение версии файла перед изменением). Если они недоступны (например, при запуске теста вне окружения), используются заглушки.
- **Строки 19–28**: Функция `_git_auto_snapshot`. Если в проекте настроен Git, она автоматически делает коммит/снимок после изменения файла.
- **Строки 31–32**: Функция `_skip_dir`. Список папок, которые агент должен игнорировать (например, `.git`, `node_modules`, `__pycache__`), чтобы не тратить время и ресурсы.

### `read_file` (Строки 36–61)
Инструмент для чтения содержимого файла.
- **Параметры**:
  - `filename`: Путь к файлу.
  - `offset` / `limit`: Позволяют читать файл по частям (например, с 100-й строки, 50 строк), что важно для очень больших файлов.
- **Логика**: Разрешает путь, читает текст (с поддержкой UTF-8), если файл бинарный — возвращает предупреждение. Возвращает словарь с контентом и метаданными (путь, сколько строк всего).

### `list_files` (Строки 65–86)
Аналог команды `ls` или `dir`.
- **Параметры**: `path` (папка), `recursive` (заходить ли в подпапки), `pattern` (фильтр, например `*.py`).
- **Код**: Использует библиотеку `pathlib` для перебора объектов. Если режим не рекурсивный, просто выводит содержимое папки. Если рекурсивный — использует `rglob` для поиска по шаблону.

### `search_in_files` (Строки 90–109)
Поиск текста внутри файлов (как `grep` или `Ctrl+Shift+F` в VS Code).
- **Логика**: Проходит по всем файлам в директории (по шаблону `file_pattern`), читает их и ищет вхождение строки `query` (без учета регистра). Ограничивает количество найденных файлов (`max_files`), чтобы не перегружать контекст LLM.

### `edit_file` (Строки 113–163) — КРИТИЧЕСКИ ВАЖНО
Основной инструмент для правки кода.
- **Логика**:
  1. Если `old_str` пустой — работает как `write_file` (создает или очищает файл).
  2. Если `old_str` указан, программа ищет его в файле.
  3. Если найдено, делает копию (snapshot) старой версии файла.
  4. Заменяет **первое** вхождение `old_str` на `new_str`.
  5. Сохраняет файл и вызывает Git-автоснимок.
- **Зачем это так сделано?** Агент заменяет только конкретные блоки, что безопаснее, чем переписывать весь файл целиком (меньше риск ошибок).

### `write_file` (Строки 167–189)
Полная перезапись файла.
- **Логика**: Создает папку (если её нет), делает бэкап существующего файла через `save_version`, записывает новый контент.

## Примеры использования

**Агент хочет прочитать конфиг:**
```python
read_file(filename="config.json")
```

**Агент хочет исправить опечатку в функции:**
```python
edit_file(
    path="main.py",
    old_str="def calculate(a,b):\n    retun a+b",
    new_str="def calculate(a, b):\n    return a + b"
)
```
